Design Section 1: Revised Architecture

Server-side sync with progressive feed population

Data Flow
1
User signs in with Spotify OAuth
Scopes: user-follow-read, user-library-read
2
Client syncs artist list → Supabase
Followed artists + saved album artists. ~40 paginated calls. Stored as artist IDs in user_artists table.
3
Edge Function: Background release sync
Runs every 15 min. Picks ~50 artists not checked recently. Calls GET /artists/{id}/albums?limit=1. Stores new releases in releases table. Full cycle in ~10 hours.
4
Client reads feed from Supabase
Joins releases with user_release_state (dismissed). Filters by release type + recency. Sorted newest first. Fast — no Spotify API calls.

Key Points

Strengths

  • Feed reads are instant — just a Supabase query
  • Spotify rate limits handled server-side, spread over time
  • Releases table is shared — if two users follow the same artist, we store the release once
  • Token stored server-side — Edge Function can refresh and call Spotify without user being online
  • Artist list sync is fast (~40 client-side calls on login)

Trade-offs

  • First-time users wait ~10 hours for a complete feed
  • Spotify tokens must be stored securely in Supabase
  • Edge Function needs scheduling (pg_cron or external trigger)
  • More Supabase infrastructure than a pure client-side approach
