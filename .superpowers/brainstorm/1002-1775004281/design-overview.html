Key Architecture Decisions

This Design

  • Follows starter's 3-layer pattern: SpotifyService → ReleasesStore → ReleasesFeed
  • Client-side Spotify calls — no backend proxy needed
  • Supabase stores only user state (dismissed IDs, preferences, last-checked)
  • Spotify catalog data always fresh — no staleness problem
  • OAuth scopes: user-follow-read + user-library-read

Trade-offs

  • Many API calls needed for large libraries (paginated, rate-limited)
  • Token refresh must be handled client-side
  • No offline/cached fallback for Spotify data
